# M7 + M8 — Conversation Engine and HiTL Handoff

Built against Mason's verbatim Conversational Selling Map (4 stages), the two
named objection reframes (Uncertainty + Overwhelm), and the 5 ghost follow-up
templates with 2-day cadence. Mason's prime directive — *"you can't automate
this because this opens up a conversation"* — drives the architecture: low
confidence thresholds, fast handoff, agent drafts but human approves the
nuanced moments.

## What's in M7

```
app/modules/m7_conversation/
├── prompts.py        # Mason verbatim content + system prompt template
├── decision.py       # Pydantic schema for AgentDecision + ValidationResult
├── selling_map.py    # Pure state machine: stage transitions
├── validator.py      # Guardrails: length, banned phrases, big-ask, confidence, high-value
├── agent.py          # Claude wrapper — context → AgentDecision
├── repository.py     # Supabase access for conversations/messages/followups/handoffs
├── io_dm.py          # Playwright IG DM poll + send
└── worker.py         # Main loop: poll → agent → validate → send → state update
```

## What's in M8

```
app/modules/m8_handoff/
├── ghost_followups.py   # Sender for the 5 Mason templates on 2/4/7/14/28-day cadence
└── discord_bot.py       # Slash commands: /claim /resolve /return /queue + new-handoff poster
```

## Decision flow per inbound

```
prospect sends DM
   ↓
io_dm.poll_inbox() — finds unread thread
   ↓
repository.insert_inbound_message() + cancel any scheduled followups
   ↓
agent.decide() — Claude returns AgentDecision JSON
   ↓
validator.validate() — runs every guardrail check
   ↓ (5 outcomes)
   ├── forced_handoff   → create handoff row + Discord embed
   ├── handoff (agent)  → same
   ├── drop             → conversation → closed_lost
   ├── hold             → no outbound, wait for next inbound
   └── send             → io_dm.send_dm() → insert_outbound_message() → update stage
```

## Validator override matrix

| Trigger | Forced handoff? | Reason |
|---|---|---|
| Prospect says "how much", "how does it work", "let's call" | ✓ | `high_value` |
| Confidence < 0.60 (reply) or < 0.75 (advance) | ✓ | `low_confidence` |
| Same objection raised twice | ✓ | `objection_escalation` |
| Message contains banned corporate phrase | ✓ | `low_confidence` (treated as agent error) |
| Calendar link in first 2 outbounds | ✓ | `low_confidence` |
| Message > 280 chars (unless verbatim reframe) | ✓ | `low_confidence` |
| Message > 3 sentences | ✓ | `low_confidence` |
| I/me > you/your ratio | flag-only | (logged, not forced) |

## Ghost follow-up cadence (Mason verbatim, 5 templates)

| # | Wait | Template |
|---|---|---|
| 1 | 48h | "No rush. Just wanted to leave this here in case it's useful." |
| 2 | 96h | "Still happy to send you the link if it helps. No pressure at all." |
| 3 | 168h (1wk) | "Hey, quick heads up. Spots are filling up for this week..." |
| 4 | 336h (2wk) | "Checking to make sure you saw this." |
| 5 | 672h (4wk) | "Coming back to this because I know we'd crush together." |

After #5, no further automated follow-up. If a prospect responds at any
point, all pending followups are cancelled (`cancelled_response`) and the
ghost counter resets.

## What does NOT exist yet (deliberate gaps)

1. **Stable IG message ID capture.** IG web doesn't expose stable IDs for
   messages. Dedupe is by `(conversation_id, direction, body)` — fragile but
   workable. If duplicates show up, harden in `_already_recorded()`.

2. **Robust thread-side direction detection.** `poll_inbox()` returns all
   message text from the visible thread; we filter by checking the DB for
   matching outbound bodies and assume everything else is inbound. This is
   right 99% of the time but can misfile if the prospect quotes our exact
   message back. Acceptable for V1.

3. **Reframes for price / time / trust / "let me think about it"** — Mason's
   sources don't script these. Tagged as NOT IN SOURCES in `prompts.py`.
   Agent is instructed to set `action="handoff"` with
   `trigger_reason="nuance_required"` when these appear. Human handles.

4. **IG DOM selectors will drift.** When poll or send fails consistently,
   `SELECTORS` in `io_dm.py` needs refresh — there's a TODO marker.

## Order to ship in production

1. Apply the .env additions on the VPS.
2. Update docker-compose.yml with the three command/service changes above.
3. Create the Discord channel; invite the bot with `applications.commands` scope.
4. `docker compose build && docker compose up -d worker_conv discord_bot worker_followups`
5. Monitor `docker logs -f fc_ig_worker_conv` for the first inbound.
6. First few handoffs WILL fire — that's the system working as designed.
   Confidence thresholds can be lowered later once you've audited the
   first ~20 conversations.
